# Задача: AI-перевод в чате (v1) с аккуратным UX и переиспользованием существующей OpenAI-интеграции

Нужно добавить в чат-приложение функцию AI-перевода через уже существующую интеграцию с ChatGPT / OpenAI в проекте.

Важно: **не изобретать новую архитектуру с нуля**, а максимально переиспользовать уже имеющиеся AI-сервисы, ключи, helper-ы, endpoints и UI-компоненты.

---

# Главная логика feature

## 1. Перевод сообщений в диалоге
Это функция для чтения переписки.

### Правило:
- **все сообщения в чате переводятся только на украинский**
- это фиксированный сценарий для v1
- выбор языка тут не нужен

### Как должно работать:
Нужна логика **как в Telegram**:
- пользователь может **включить перевод**
- после этого может **смотреть либо оригинал, либо перевод**
- должно быть удобно переключаться между этими режимами

### Важно:
Не делать грубую подмену текста без контроля.
Нужно реализовать понятный UX, где пользователь осознанно понимает:
- сейчас он смотрит **оригинал**
- или сейчас он смотрит **перевод**

---

## 2. Перевод текста, который пишет оператор
Это функция-помощник для compose/input.

### Правило:
- текст, который оператор сам набрал, как правило нужно переводить **на английский**
- для v1 считаем это основным сценарием
- здесь тоже не нужен сложный выбор языков

### Как должно работать:
Оператор пишет текст в composer → нажимает кнопку перевода → получает **английскую версию** своего текста.

---

# Архитектурный принцип

## Перед реализацией обязательно:
1. Проверить, какие OpenAI / ChatGPT / AI service / helper / HTTP wrapper / action classes / endpoints уже есть в проекте.
2. Максимально переиспользовать существующую инфраструктуру.
3. Не делать новую сложную AI-подсистему, если уже есть рабочая интеграция.
4. Не перепридумывать backend flow, если можно аккуратно встроить новый use-case в существующий.

## Важно:
Нужен **feature-level implementation**, а не новая AI-платформа.

# Сценарий A — перевод сообщений в чате на украинский

## Цель
Сделать режим просмотра переведённого диалога, где сообщения можно смотреть **как оригинал или как перевод**, по аналогии с UX Telegram.

---

## UX-логика

### В header чата нужен action:
Например:
- `Переклад UA`
- или компактный toggle / action button
- или action с иконкой + tooltip

### После активации:
Нужно не просто показать маленькие подписи, а дать понятный режим просмотра:

#### Вариант поведения для v1:
- включаешь перевод
- сообщения получают украинскую версию
- пользователь может **переключать отображение**
  - `Оригінал`
  - `Переклад`

### Очень важно:
Ориентир по UX — **как в Telegram**, где перевод не ощущается как костыль, а как отдельный режим просмотра.

---

## Как именно это должно выглядеть

Нужно продумать **чистый и понятный вариант**:

### Предпочтительный подход для v1:
Глобальный toggle на уровне текущего диалога:
- режим `Оригінал`
- режим `UA переклад`

То есть:
- если выбран `Оригінал` — показываем исходные тексты сообщений
- если выбран `UA переклад` — показываем украинские переводы вместо исходного текста в bubble
- при этом оригинал не теряется логически, а просто скрывается в текущем режиме просмотра

### Дополнительно:
Можно сохранить возможность быстро вернуться к оригиналу без повторного запроса.

---

## Анимация
Переключение между режимами должно быть:
- мягким
- быстрым
- production-friendly
- без тяжелых эффектов

Подход:
- fade transition
- slight vertical motion
- subtle content swap
- без “театральности”

---

## Важные требования
- не слать перевод на каждый ререндер
- не делать бесконечные повторные AI-запросы
- перевод загружать **только по явному действию**
- переводить только **текущий открытый чат**
- язык перевода для диалога всегда `uk`

---

## Хранение на фронте
Для v1 желательно:
- хранить переводы локально по `message_id`
- если перевод уже был получен, использовать кеш
- не дёргать API повторно без необходимости

---

# Сценарий B — перевод текста оператора в composer на английский

## Цель
Дать оператору быстрый помощник перевода перед отправкой сообщения.

### Правило:
Текст из composer переводится **на английский**.

---

## UX
Внутри composer нужен action:
- кнопка `Translate to EN`
- или иконка с tooltip
- или компактный secondary action

### При нажатии:
1. Берётся текущий текст из input / textarea
2. Отправляется на перевод
3. Возвращается английская версия
4. Пользователь получает результат в удобной форме

---

## Предпочтительный UX для v1
Самый понятный вариант:
- после перевода показать маленький preview-блок над composer
- внутри:
  - английская версия текста
  - кнопка `Replace draft`
  - кнопка `Cancel`

### Почему так лучше:
- не будет внезапной подмены текста
- пользователь контролирует действие
- UX выглядит зрелым и понятным

### Допустимый упрощённый вариант:
Если в текущем интерфейсе preview вставить сложно — можно сразу заменять черновик, но только если это хорошо вписывается в существующий composer UX.

---

## Состояния
- если поле пустое — кнопка disabled
- если идёт запрос — локальный loading state
- если ошибка — аккуратный inline error или unobtrusive toast
- не ломать composer layout

---

- где находится toggle перевода
- как пользователь понимает, что сейчас смотрит:
  - оригинал
  - или перевод
- как показать loading на уровне всего чата

### Важно:
Режим просмотра должен быть понятен с первого взгляда.

---

## Для composer-перевода
Продумать:

- её состояние при пустом тексте
- loader
- preview translated draft
- replace / cancel flow
- чтобы панель ввода не стала перегруженной

## Принцип
Не строить новый сервис с нуля.

Нужно:
- найти текущий AI/OpenAI service в проекте
- использовать существующий client / config / auth / helper / wrapper
- добавить отдельный use-case для **translation**

Примерно:
- вход: строка черновика

# Качество перевода

## Для перевода сообщений в чате → украинский
Перевод должен быть:
- понятным
- естественным
- не литературно-пафосным
- подходящим для support/chat context

## Для перевода черновика → английский
Перевод должен быть:
- естественным
- рабочим
- понятным для клиента
- без избыточно сложных формулировок
- ближе к нормальному customer support English

---

# Edge cases

Нужно предусмотреть:
- сообщение уже на украинском
- текст уже на английском
- очень короткие сообщения: `ok`, `yes`, `try now`
- пустой composer
- слишком длинный текст
- timeout / AI error
- повторный клик во время загрузки
- повторное открытие чата, где переводы уже были получены

Поведение должно быть спокойным:
- маленький loader
- понятный fallback
- без ломания чата
- без агрессивных error states

---
